# Sub2API Multi-Site Menu Bar Refresh Design

## 1. 背景

本设计用于收敛 `Sub2API` 菜单栏应用第一版的最终交互与视觉细节，重点补齐此前未定稿的多站点能力、窄弹层布局和账号卡信息密度。

本文件是后续实现与计划的主依据。若与旧版设计冲突，以本文件为准。

## 2. 本轮确认后的目标

第一版产品仍然是一个：

`面向 Sub2API 管理员的 macOS 菜单栏监控器`

它只解决三件事：

1. 不打开网页时，快速看到当前站点的整体消耗
2. 快速判断账号池里哪些账号还能继续轮询、哪些已经接近或达到窗口上限
3. 在有多个 `sub2api` 站点时，能直接切换到不同站点查看同类信息

非目标保持不变：

- 不做原生后台管理系统
- 不做账号详情窗口
- 不做每个账号单独跳转
- 不做重告警样式
- 不做跨站点聚合报表

## 3. 核心结论

### 3.1 多站点

- 应用支持维护多个站点配置
- 菜单栏弹层一次只展示一个当前站点
- 顶部站点选择器默认收起，点击后以下拉列表形式展开
- 切换站点后立即刷新当前站点数据
- 设置页负责站点的新增、编辑、删除与默认站点切换

### 3.2 菜单栏弹层

- 弹层继续保持此前用户认可的窄宽度，不做宽桌面面板
- 目标宽度控制在 `400 - 420px`
- 内容全部采用上下堆叠，避免网页后台式的左右并排
- 首屏默认应能看到顶部摘要与至少两张账号卡片

### 3.3 顶部结构

- 顶部不显示“弹层页签”这类说明性文字
- 只保留实际可操作内容：
  - 当前站点下拉
  - `账号监控` / `仪表盘` Tab
  - `设置`
  - `刷新`

### 3.4 仪表盘

- 仪表盘保留为原生 `SwiftUI` 页面
- 只展示站点摘要与模型分布
- 移除“用户消费榜”
- 布局保持窄弹层友好，不嵌入网页式双栏结构

## 4. 信息架构

第一版最终信息架构固定为：

1. 菜单栏弹层
2. 设置窗口
3. 浏览器打开当前站点

其中菜单栏弹层包含两个一级 Tab：

- `账号监控`
- `仪表盘`

## 5. 菜单栏弹层设计

### 5.1 头部

头部区域从左到右为：

1. 当前站点下拉按钮
2. `设置`
3. `刷新`

当前站点下拉按钮的行为：

- 默认展示当前站点名称与下拉箭头
- 默认收起列表
- 点击后展开一个轻量下拉面板
- 下拉内仅展示站点名称、当前选中状态和可点击切换行为
- 不在这里放编辑表单

头部只做站点上下文切换和全局动作，不承载大块错误面板。

### 5.2 一级 Tab

Tab 采用扁平 segmented control 风格：

- `账号监控`
- `仪表盘`

要求：

- 切换时不改变总宽度
- 选中态清晰但不过度强调
- 不额外显示二次标题文案

### 5.3 账号监控 Tab

`账号监控` 是默认首页。

页面结构固定为：

1. 站点总览卡
2. 轻量筛选行
3. 账号卡片列表
4. 底部统一动作

#### 5.3.1 站点总览卡

站点总览卡采用紧凑的上下分区，不拆成多个横向大卡。

必须展示：

- `今日 Token`
- `总 Token`
- `今日金额`
- `总金额`

金额信息放入 Token 信息右侧或下方紧凑区域，用来填补此前留白，但整体仍保持窄弹层可读性。

不再展示：

- 今日请求数

如接口可提供补充摘要，可在卡片底部追加一行轻量信息：

- 活跃模型数
- 当前主消耗模型

这类信息必须是次级视觉，不得挤占 Token 与金额的主位置。

#### 5.3.2 筛选

保留轻量筛选，用于快速切换账号集合：

- `全部`
- `正在使用中`
- `5h 高占用`
- `不可用`

规则：

- “正在使用中”结果允许为空
- 没有命中时显示轻量空状态文案，不制造假数据
- 筛选只影响当前列表，不影响站点原始数据

#### 5.3.3 账号卡片列表

账号区域采用单列紧凑卡片堆叠。

目标：

- 一屏至少看到两张账号卡片
- 单卡高度明显低于当前实现中的大卡版本
- 允许纵向滚动

每张卡片必须展示：

- 账号名称
- 当前状态
- 正在使用中或最近使用信息
- 当前或最近使用模型
- `5h` 占用进度
- `5h` 倒计时
- `5h` 用量窗口摘要
- `7d` 占用进度
- `7d` 倒计时
- `7d` 用量窗口摘要

卡片布局原则：

- 顶行：账号名 + 状态胶囊
- 次行：使用状态、并发、最近使用、模型信息
- 中段：`5h` 与 `7d` 两组紧凑进度条
- 底部：两组窗口摘要，使用小号文字或 pill 表达

不提供：

- 原生详情页
- 账号卡点击跳转
- 账号级操作按钮

#### 5.3.4 窗口摘要表达

`5h` 和 `7d` 的窗口摘要采用同一格式，目标是让用户一眼判断还能不能继续轮询。

每组摘要至少包含：

- 已用占比
- 剩余可用
- 剩余倒计时或重置时间

表达方式应尽量扁平，不做大块解释区域。

#### 5.3.5 底部动作

底部只保留一个统一入口：

- `在浏览器中打开站点`

点击后打开当前选中站点的网页后台首页。

### 5.4 仪表盘 Tab

仪表盘也保持窄弹层结构，所有信息上下排列。

页面结构固定为：

1. 顶部摘要卡
2. 模型分布列表

顶部摘要卡至少展示：

- 今日实际金额
- 今日标准金额
- 活跃模型数
- 当前主消耗模型

模型分布列表每行展示：

- 模型名称
- Token
- 金额
- 占比

不再展示：

- 用户消费榜

## 6. 设置窗口设计

设置窗口继续使用扁平、浅色、轻量的设计语言，与弹层保持一致。

### 6.1 站点管理方式

设置页负责多站点管理，而不是仅配置单站点表单。

固定结构：

1. 站点列表区
2. 当前站点编辑表单
3. 测试与保存动作区

需要支持：

- 新增站点
- 编辑站点名称
- 编辑站点地址
- 编辑 Admin Token
- 删除站点
- 选定默认站点

### 6.2 字段

每个站点配置包含：

- 站点名称
- 站点地址
- Admin Token

### 6.3 Token 指引

`Admin Token` 输入区下方增加明确指引：

`不知道在哪找？点击查看获取方式`

展开后展示：

1. 打开 `sub2api` 管理后台
2. 浏览器按 `F12` 打开控制台
3. 执行 `localStorage.getItem('auth_token')`

### 6.4 按钮

设置页底部保留两个主要动作：

- `测试连接`
- `保存并刷新`

行为要求：

- `测试连接` 只验证当前表单配置是否可用，并反馈成功或失败
- `保存并刷新` 负责持久化当前站点配置，并触发主界面刷新

### 6.5 本地存储

用户已明确不希望应用访问系统钥匙串，因此第一版不使用 Keychain。

配置本地保存即可，但需要满足：

- 下次启动可恢复站点列表
- 能记住当前默认站点

## 7. 数据规则

### 7.1 站点粒度

- 当前界面只展示一个站点的数据
- 不做跨站点汇总
- 切换站点时，标题、摘要、账号卡、仪表盘都应同步切换

### 7.2 `5h / 7d` 数据来源

`5h / 7d` 相关展示应优先与 `sub2api` 网页端保持一致。

首选数据来源：

- 账号列表里的 `extra.codex_5h_*`
- 账号列表里的 `extra.codex_7d_*`

仅在上述字段缺失时，才允许回退到单账号使用窗口接口。

原因：

- 网页端实际已优先依赖这些字段
- 某些账号的窗口接口可能返回失败，不能把它作为唯一来源

### 7.3 正在使用中判断

“正在使用中”按以下规则判断：

- `currentConcurrency > 0`
- 或 `lastUsedAt` 距当前时间不超过 `10` 分钟

若没有账号满足条件，“正在使用中”筛选允许为空。

### 7.4 默认排序

默认排序目标只有两个：

1. 正在使用的账号排前面
2. 不可用账号排后面

推荐分组顺序：

1. `正在使用中`
2. `可用且高占用`
3. `可用且普通`
4. `不可用`

同组内再按以下顺序：

1. 最近使用时间更近
2. `5h` 占用更高
3. 名称字母序

### 7.5 告警表达

问题账号仍然需要被看见，但不采用夸张告警。

允许使用：

- 状态胶囊
- 进度条颜色
- 小号提示文案

不使用：

- 大块红底横幅
- 闪烁效果
- 抢占版面的错误卡

## 8. 视觉约束

风格继续尽量还原 `sub2api` 的浅色、扁平、克制感。

固定约束：

- 白底或极浅灰底
- 细描边
- 小圆角
- 弱阴影或无阴影
- 青绿系为主强调色
- 视觉层级靠留白和字重，不靠厚重装饰

设计关键词：

- 扁平
- 紧凑
- 清爽
- 信息优先
- 菜单栏友好

## 9. 验收标准

实现阶段至少需要验证：

- 顶部站点列表默认收起，点击后以下拉形式展开
- 多站点切换后能刷新并切换到对应站点数据
- 弹层宽度保持在既定窄宽度，不再出现过宽面板
- `账号监控` 首屏至少能看到两张账号卡片
- 顶部总览能同时清楚展示 Token 与金额
- 单卡内能同时看见模型信息、`5h / 7d` 进度、倒计时和窗口摘要
- `5h / 7d` 与网页端核心数据保持一致
- 不可用账号不会排到列表前部
- 仪表盘不再展示用户消费榜
- 设置页可完成 `测试连接` 与 `保存并刷新`
- 应用不再触发系统钥匙串访问

## 10. 最终结论

第一版最终形态应是一个：

`支持多站点切换、保持窄弹层、以账号卡片监控为主的 Sub2API 原生菜单栏应用`

它承担“高频观察入口”的职责，而不是替代网页后台。
