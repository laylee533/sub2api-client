# Sub2API 像素风图标设计

日期：2026-04-13

## 目标

为 `Sub2APIMonitorApp` 补齐缺失的应用图标与菜单栏图标，风格保持：

- 像素风
- 扁平化
- 监控工具语义明确
- 与当前浅色、克制的 `sub2api` 风格兼容

本次只做一套主方向，不做多主题切换，不做动态图标。

## 已确认设计

### 1. App 图标

采用 `Pixel Monitor` 方向。

核心形态：

- 一个像素化 CRT 监视器
- 深色机身
- 绿色终端数据条
- 右上角一个琥珀状态灯
- 浅灰蓝底色，保持清爽，不走高饱和 neon 风

表达语义：

- 这是一个“账号监控器”
- 不是通用终端工具，也不是复杂控制台
- 重点是“正在观察多个账号状态”

### 2. 菜单栏图标

采用 `A1 单色模板版`。

核心形态：

- 保留监视器轮廓
- 保留屏幕内两到三段像素条
- 去掉彩色状态灯
- 作为单色 template image 使用，适配 macOS 顶栏明暗模式

表达语义：

- 在顶栏小尺寸下优先保证识别度
- 不强调告警，不通过红黄绿状态强提醒
- 与用户“报警不要太明显”的偏好一致

## 视觉规范

### 图标语言

- 8-bit / 低分辨率像素块感
- 边缘清晰，不做柔和渐变描边
- 扁平、克制，不做玻璃拟物
- 允许少量景深式底色渐变，但主体保持硬边像素感

### 建议配色

- 背景：`#DCE6EC`
- 外框深色：`#0F172A`
- 外框次深色：`#334155`
- 屏幕底色：`#0E2B21`
- 主数据绿：`#34D399`
- 次数据绿：`#10B981`
- 状态灯：`#F59E0B`
- 菜单栏模板色：跟随系统 template tint，不自带颜色

### 视觉边界

- App 图标允许一个琥珀状态灯作为识别点
- 菜单栏图标不保留彩色状态灯
- 不加入文字、字母、数字
- 不加入过多账号槽位细节，避免缩小后糊成噪点

## 交付物

### 1. App 图标资源

输出一套 macOS 可用的应用图标资源，用于：

- Finder
- Launch Services
- DMG 中的 `.app`

推荐交付形式：

- 一个可维护的源文件
- 一组导出的 `iconset`
- 最终 `.icns`

源文件应优先保持可重复导出，而不是只保留最终位图。

### 2. 菜单栏图标资源

输出一份单色模板图标资源，用于：

- `MenuBarExtra` 标签图标
- 顶栏高亮、浅色、深色场景

该资源必须满足：

- 小尺寸下轮廓清晰
- 不依赖彩色信息
- 可被标记为 template image

## 技术落地方案

### App 图标接入

当前项目是 `Swift Package Manager + 自定义 app bundle 打包脚本`，没有现成 `Assets.xcassets`。

因此 App 图标落地方式为：

1. 在仓库内新增图标源文件与导出产物
2. 打包时把 `.icns` 复制到 `Sub2APIMonitorApp.app/Contents/Resources/`
3. 在 `Info.plist` 中补充 `CFBundleIconFile`

这样既兼容当前脚本式打包流程，也不要求先迁移到完整 Xcode App 工程。

### 菜单栏图标接入

当前菜单栏入口使用：

```swift
Label("sub2api 监控器", systemImage: model.statusItemSymbolName)
```

后续改为：

- 使用自定义 `NSImage`
- 将该图标标记为 template image
- 在 `MenuBarExtra` 中用自定义图标替代 `SF Symbol`

初版不做多状态图标切换。

理由：

- 用户关注的是账号卡片内容，不是菜单栏上的强状态提示
- 顶栏图标过度变化会增加噪声
- 单色模板最稳定，最符合 macOS 习惯

## 资源组织建议

推荐新增以下内容：

- 一个图标源目录，保存像素源稿与导出脚本
- 一个运行时资源目录，保存菜单栏模板图标
- 一个打包资源目录，保存最终 `.icns`

资源应避免只保留不可追溯的最终文件。

## 验证方式

### 自动验证

- `swift build`
- `swift test`
- `./scripts/package-dmg.sh`

### 手动验证

- Finder 中 `.app` 显示自定义 App 图标
- 安装到 `Applications` 后显示相同图标
- 菜单栏中出现单色像素监视器图标
- 暗色模式和亮色模式下都清晰可辨
- 菜单栏图标不会因为 template 渲染而丢失主体结构

## 风险与取舍

### 1. 菜单栏尺寸过小

风险：

- 像素风细节一多就会糊

取舍：

- 菜单栏图标只保留最核心的轮廓与两到三段内部条形

### 2. App 图标与菜单栏图标风格不完全同构

风险：

- App 图标更丰富，菜单栏图标更简化

取舍：

- 允许同一视觉语言下的“复杂版 / 极简版”配套
- 优先保证不同使用场景的可读性，而不是强行完全一致

### 3. 当前打包流程不包含图标资源管理

风险：

- 若只补视觉资源而不改打包脚本，最终 `.dmg` 仍可能没有图标

取舍：

- 实现阶段必须同步修改打包脚本与 `Info.plist` 生成逻辑

## 本次不做

- 按账号状态动态切换菜单栏图标
- 红黄绿告警版菜单栏图标
- 动画图标
- Dock 徽章数字
- 多主题图标
