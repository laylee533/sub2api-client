# Sub2API macOS Menu Bar Monitor Design

## 1. 背景与目标

本项目计划基于 `sub2api` 构建一个 macOS 原生应用，技术栈为 `Swift + SwiftUI`。

该应用的目标不是替代 `sub2api` 网页后台，而是提供一个更轻量、更高频可用的账号池观察入口，让管理员无需频繁打开网页，也能快速判断当前账号池是否还能继续轮询工作。

本次设计最终聚焦于：

- 管理员场景，而不是普通用户场景
- `sub2api` 管理端账号池监控，而不是用户 API Key 看板
- 菜单栏弹层作为主入口
- `sub2api` 控制台风格的原生扁平化界面
- 快速查看账号状态，不承载复杂后台编辑

## 2. 产品定位

产品定位为：

`Sub2API 管理员账号池菜单栏监控器`

核心价值：

- 打开菜单栏即可看到账号池当前状态
- 优先看到正在使用中的账号
- 一眼看出各账号 `5h / 7d` 窗口占用与倒计时
- 通过站点级总览快速掌握整体运行热度
- 当需要更完整信息时，统一跳转到 `sub2api` 网页端

非目标：

- 不做完整的原生后台管理面板
- 不做原生账号编辑、导入导出、复杂筛选
- 第一版不做原生详情窗口
- 第一版不要求逐账号深链接到网页端单账号详情

## 3. 目标用户

目标用户是 `sub2api` 管理员。

典型使用场景：

- 菜单栏点开后快速查看“当前哪些账号正在使用”
- 观察哪些账号 `5h` 快耗尽、哪些账号已限流
- 观察 `7d` 窗口是否逼近上限
- 发现异常后决定是否打开网页端进一步处理

## 4. 信息架构

第一版信息架构收敛为两层：

1. 菜单栏弹层
2. 网页端跳转

### 4.1 菜单栏弹层

菜单栏弹层是主界面，承担以下职责：

- 显示当前站点总览
- 显示账号卡片矩阵
- 提供轻量筛选
- 提供统一网页跳转入口

### 4.2 网页端跳转

网页端仅作为深度信息与复杂操作承载体。

第一版只保留底部统一入口：

- 在浏览器中打开站点
- 打开账号列表页

说明：

- 当前上游前端已存在 `/admin/accounts` 页面
- 暂未确认存在稳定可依赖的“单账号详情路由”
- 因此第一版不绑定逐账号网页深链接，避免做出脆弱跳转逻辑

## 5. 菜单栏弹层布局

菜单栏弹层推荐宽度：

- `420px - 480px`

推荐高度：

- 首屏尽量控制在单次滚动内可浏览的范围
- 当账号较多时允许纵向滚动

布局分为四段：

### 5.1 站点头部

展示：

- 站点名称
- 账号总数
- 手动刷新按钮

用途：

- 明确当前查看的是哪个 `sub2api` 站点
- 提供最小刷新能力

### 5.2 站点总览指标

顶部总览固定显示站点级指标：

- 今日请求
- 今日 Token
- 总 Token

这些指标是全站汇总，不属于单个账号卡片。

设计原因：

- 用户明确要求这里显示“总的”，不是每个账号的消耗
- 这三项足以表达当前站点总体热度

### 5.3 轻量筛选条

筛选条作为横向 chip 列表，建议支持：

- 正在使用中
- 5h 高占用
- 异常
- 全部

筛选只影响当前卡片排序和展示集合，不进入复杂查询流程。

### 5.4 账号卡片矩阵

使用双列卡片矩阵。

卡片是主信息载体，每张卡片只展示账号本身相关信息，不展示站点总量。

每张卡片展示：

- 账号名称
- 状态标签
- 当前并发或活跃提示
- `5h` 窗口占用百分比
- `5h` 重置倒计时
- `7d` 窗口占用百分比
- `7d` 重置倒计时
- 最近使用时间

不展示：

- 原生详情按钮
- 单卡片跳网页按钮
- 卡片级今日请求
- 卡片级今日 Token
- 卡片级总 Token

说明：

- 卡片级今日请求在设计过程中被移除，避免与顶部站点总览重复
- 第一版卡片强调“窗口状态”和“当前能否继续轮询”

### 5.5 底部统一动作区

底部保留统一入口：

- 打开账号列表页
- 在浏览器中打开站点

统一入口比卡片级入口更干净，也更符合用户“只是为了不需要经常去打开网页”的目标。

## 6. 排序与优先级

默认排序规则：

1. 正在使用中的账号
2. `5h` 高占用账号
3. 异常或限流账号
4. 其余账号

“正在使用中”的判断规则：

优先使用：

- `current_concurrency > 0`

回退判断：

- `last_used_at` 非常接近当前时间

原因：

- 用户明确要求优先显示正在使用中的账号
- 对管理员来说，这些账号是最值得优先观察的运行面

## 7. 状态映射

UI 不直接暴露所有后端原始字段，而是映射为较稳定的展示状态：

- `in_use`
- `healthy`
- `tight`
- `rate_limited`
- `unschedulable`

建议映射逻辑：

- `in_use`：当前并发大于 0
- `rate_limited`：存在限流状态或限流恢复时间
- `unschedulable`：不可调度、错误、临时不可调度
- `tight`：`5h` 或 `7d` 占用达到阈值
- `healthy`：其余正常状态

推荐阈值：

- `5h >= 80%` 视为紧张
- `7d >= 80%` 视为紧张

## 8. 数据模型

建议客户端建立面向 UI 的聚合模型，而不是直接把后端返回结构塞进视图。

### 8.1 SiteSummary

字段建议：

- `siteName`
- `baseURL`
- `totalAccounts`
- `todayRequests`
- `todayTokens`
- `totalTokens`
- `lastRefreshedAt`

### 8.2 AccountCardModel

字段建议：

- `id`
- `name`
- `platform`
- `type`
- `state`
- `currentConcurrency`
- `lastUsedAt`
- `fiveHourUtilization`
- `fiveHourResetAt`
- `fiveHourCountdownText`
- `sevenDayUtilization`
- `sevenDayResetAt`
- `sevenDayCountdownText`
- `sortPriority`

### 8.3 FilterMode

字段建议：

- `active`
- `high5h`
- `abnormal`
- `all`

## 9. 数据来源与接口映射

第一版数据主要来自 `sub2api` 管理端现有接口：

- `/api/v1/admin/accounts`
- `/api/v1/admin/accounts/today-stats/batch`
- 需要时可补充 `/api/v1/admin/accounts/:id/usage`

设计上的接口使用建议：

### 9.1 账号列表

用于获取：

- 基础账号信息
- 状态
- 调度能力
- 最近使用
- 实时并发

### 9.2 批量今日统计

用于补充：

- 账号今日请求
- 今日 Token
- 今日成本

但在第一版最终界面中，这些账号级统计不必全部展示在卡片上，可作为内部排序或未来扩展数据保留。

### 9.3 站点总览计算

当前站点总览的三项：

- 今日请求
- 今日 Token
- 总 Token

可由批量统计与账号使用统计聚合得到。

若后端后续提供更直接的站点汇总接口，可替换为汇总接口，减少客户端计算成本。

## 10. 刷新策略

为了让菜单栏足够实用，刷新策略建议为：

- 打开弹层时立即刷新
- 弹层保持展开时每 `20-30` 秒自动刷新
- 用户点击刷新按钮时立即触发手动刷新
- App 后台常驻时可定时预热缓存，但不要求高频轮询

刷新目标：

- 保证“正在使用中”排序尽可能及时
- 保证 `5h / 7d` 倒计时和占用状态不明显陈旧

## 11. 视觉风格

视觉方向明确要求：

- 尽量还原 `sub2api` 控制台风格
- 同时进一步扁平化，适应 macOS 菜单栏弹层

具体风格原则：

- 主色使用 `teal/cyan` 系
- 白底或极浅灰底
- 细边框
- 轻阴影或无阴影
- 小面积状态色点缀
- 不做厚重后台布局
- 不做复杂渐变和强烈装饰

卡片风格关键词：

- 扁平
- 紧凑
- 信息优先
- 状态清晰

## 12. 导航与交互

第一版交互尽量简单：

- 点菜单栏图标打开弹层
- 点筛选 chip 切换卡片集合
- 点刷新按钮重新拉数据
- 点底部按钮统一跳网页端

不做：

- 原生多层导航
- 原生详情页
- 卡片内复杂操作菜单

## 13. 错误处理

必须覆盖的异常状态：

- 站点不可达
- 登录态失效
- 接口超时
- 部分数据拉取成功、部分失败

UI 行为建议：

- 顶部展示简短错误提示，不要用大面积告警面板
- 保留最后一次成功缓存，避免弹层空白
- 用户可手动刷新重试
- 登录失效时引导重新登录网页端或重新授权客户端

## 14. 测试与验证建议

设计阶段建议后续实现至少验证：

- 菜单栏弹层在不同账号数量下的排版稳定性
- `5h / 7d` 倒计时文本更新正确
- “正在使用中优先”排序正确
- 站点总览聚合正确
- 刷新失败时缓存兜底正常
- 网页端跳转按钮能够稳定打开目标站点

## 15. 后续实现范围建议

建议第一阶段只做：

- 多站点配置与切换
- 菜单栏弹层 UI
- 站点总览
- 账号卡片矩阵
- 轻量筛选
- 统一网页跳转
- 自动刷新与缓存

建议暂缓：

- 原生详情窗口
- 卡片级操作按钮
- 原生恢复状态/清限流
- 原生批量操作
- 复杂图表

## 16. 最终结论

第一版产品应实现为：

`一个以菜单栏弹层为主入口的 Sub2API 管理员账号池监控器`

它的核心职责只有三件事：

- 用站点总览展示当前整体热度
- 用账号卡片矩阵展示当前哪些账号在跑、哪些窗口接近上限、多久重置
- 在需要更完整信息时，统一跳转到 `sub2api` 网页端

这使得 App 的职责足够清晰：

- 它不是后台替代品
- 它是一个更顺手的、低打断的观察入口
