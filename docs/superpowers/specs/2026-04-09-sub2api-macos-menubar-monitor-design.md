# Sub2API macOS Menu Bar Monitor Design

## 1. 背景与目标

本项目基于 `sub2api` 构建一个 `macOS + Swift + SwiftUI` 的原生菜单栏应用。

它不是网页后台的替代品，而是一个更低打断、更高频使用的观察入口，用来让管理员在不打开网页的情况下，快速判断账号池当前是否健康、哪些账号还可继续轮询、哪些账号的 `5h / 7d` 容量已经逼近上限。

本轮设计已经收敛出的核心目标：

- 主入口固定为菜单栏弹层
- 信息展示优先于复杂操作
- 优先看账号池中“正在使用”和“快到上限”的账号
- 保持 `sub2api` 控制台的浅色、扁平、克制风格
- 需要更完整信息时，只提供统一的网页端入口

## 2. 产品定位

产品定位为：

`Sub2API 管理员账号池菜单栏监控器`

第一版只解决两个问题：

1. 在菜单栏里直观看到站点整体消耗
2. 在菜单栏里直观看到每个账号当前还能不能继续跑

非目标：

- 不做原生后台管理系统
- 不做原生账号详情窗口
- 不做卡片级网页跳转
- 不做重告警面板或复杂通知流
- 不把网页仪表盘直接塞进 WebView

## 3. 最终信息架构

第一版最终信息架构如下：

1. 菜单栏弹层
2. 设置窗口
3. 浏览器打开站点

### 3.1 菜单栏弹层

菜单栏弹层是高频主入口，顶部包含两个一级 Tab：

- `账号监控`
- `仪表盘`

### 3.2 设置窗口

设置窗口只负责站点配置与连接验证，不承担监控信息展示。

### 3.3 浏览器跳转

当用户需要查看完整后台信息时，只在弹层底部保留统一入口：

- `在浏览器中打开站点`

不再保留每个账号单独跳转，也不做原生详情页。

## 4. 菜单栏弹层总布局

弹层应保持“之前较合适的紧凑尺寸”，不要做成接近常规桌面面板的大宽窗。

建议尺寸原则：

- 宽度以 `400 - 420px` 为目标
- 内容改为上下堆叠，避免横向铺太开
- 首屏默认应能稳定看到顶部概览和至少两张账号卡片
- 账号较多时允许纵向滚动

弹层总结构分为四段：

1. 站点头部
2. 一级 Tab
3. 当前 Tab 内容
4. 底部统一动作区

### 4.1 站点头部

头部固定展示：

- 当前站点名称
- 账号总数
- `设置`
- `刷新`

顶部错误提示如需出现，应采用轻量行内提示，不使用强烈告警块抢占主要视觉。

### 4.2 一级 Tab

使用扁平化分段切换样式，在同一弹层内切换：

- `账号监控`
- `仪表盘`

切换后弹层宽度保持不变，只更新内容区。

## 5. 账号监控 Tab

`账号监控` 是第一优先级界面，也是菜单栏弹层默认展示的 Tab。

### 5.1 顶部站点汇总卡

顶部保留两张紧凑汇总卡：

- `今日 Token`
- `总 Token`

每张卡片采用左右分栏，但整体仍保持紧凑：

- 左侧：主数值，展示 Token 数
- 右侧：金额信息，展示 `实际金额 / 标准金额`

这样可以在不增加额外宽度的情况下，把用户最关心的金额信息补进来，并填满之前右侧留白。

### 5.2 轻量筛选

汇总卡下方保留轻量筛选行，用于快速切换关注集合。推荐筛选项：

- `全部`
- `正在使用中`
- `5h 高占用`
- `不可用`

筛选仅影响当前展示集合与排序，不引入复杂查询体验。

### 5.3 账号列表布局

账号区域采用：

- 单列紧凑卡片堆叠
- 卡片高度压缩
- 间距较小
- 一屏至少可展示两张账号卡

不再使用早期的双列矩阵方案，因为那会迫使整体弹层变宽，不符合最终确认的菜单栏尺寸要求。

### 5.4 单个账号卡片内容

每张账号卡片必须保留以下信息：

- 账号名称
- 当前状态
- 正在使用中或最近使用信息
- `5h` 占用进度
- `5h` 倒计时
- `7d` 占用进度
- `7d` 倒计时
- `5h` 用量窗口信息
- `7d` 用量窗口信息

卡片内的信息布局原则：

- 第一行展示账号名与状态胶囊
- 第二行展示“并发中”或“最近使用”
- `5h` 与 `7d` 分别使用紧凑的横向进度条
- 每个窗口下方补一行超紧凑的窗口信息，不做大块说明区

### 5.5 用量窗口信息表达

本轮确认采用“更紧凑的窗口摘要”方案。

每个窗口显示为一组小尺寸文本或 pill，推荐包含：

- 已用占比
- 剩余可用
- 重置时间点或剩余倒计时

目标是让用户“一眼知道这个窗口还能不能继续跑”，而不是展开详细报表。

### 5.6 卡片交互

单个账号卡片不提供：

- 点击进入详情
- 点击跳转网页
- 重操作按钮

这样可以保证卡片足够紧凑，也符合“这个 app 只是为了不需要经常去打开网页”的目标。

### 5.7 底部动作区

`账号监控` Tab 底部只保留一个统一动作：

- `在浏览器中打开站点`

它用于进入完整的 `sub2api` 网页后台，承接所有深度查看和后续操作。

## 6. 仪表盘 Tab

`仪表盘` 采用原生 `SwiftUI` 实现，不嵌入网页，也不直接复刻整个后台页面布局。

它的作用不是替代网页端，而是在同一个窄弹层里补充更偏“汇总趋势”的信息。

### 6.1 总体布局

`仪表盘` Tab 也必须保持与 `账号监控` 相同的窄弹层宽度。

布局原则：

- 所有内容竖向堆叠
- 不使用网页式左右双栏
- 通过分段、卡片和列表表达信息

### 6.2 子 Tab

`仪表盘` 内部保留两个二级切换：

- `模型分布`
- `用户消费榜`

二级切换建议采用紧凑 segmented control 或扁平 chip。

### 6.3 仪表盘顶部摘要

在二级内容前，展示一组轻量摘要信息，重点补充账号监控页之外的金额与模型视角。

应至少覆盖：

- 今日实际金额
- 今日标准金额
- 主要活跃模型数
- 当前最主要消耗模型

信息表现保持扁平紧凑，不单独放大告警。

### 6.4 模型分布

`模型分布` 子页重点展示：

- 模型名称
- 占比
- 对应 Token
- 对应金额

形式建议为窄列表或迷你条形图列表，每一项一行，避免横向展开。

### 6.5 用户消费榜

`用户消费榜` 子页重点展示：

- 账号或用户名称
- 今日消耗金额
- 今日 Token
- 主用模型

按消费由高到低排序，保留前若干名即可。

## 7. 设置窗口

设置窗口采用此前确认的 `A` 版方向：

- 轻量扁平
- 与主弹层同一视觉语言
- 信息块清晰
- 交互重心集中在连接配置

### 7.1 表单字段

第一版设置窗口保留：

- 站点名称（可选）
- 站点地址
- Admin Token

### 7.2 Token 获取指引

在 `Admin Token` 输入区下方增加明确指引：

`不知道在哪找？点击查看获取方式`

展开后显示：

1. 打开 `sub2api` 管理后台
2. 浏览器按 `F12` 打开控制台
3. 执行 `localStorage.getItem('auth_token')`

要求：

- 指引足够直白
- 不需要用户阅读额外文档
- 可复制代码片段

### 7.3 按钮区

设置窗口底部保留两个主要动作：

- `测试连接`
- `保存并刷新`

其中：

- `测试连接` 只验证站点地址和 token 是否可用，并回显结果
- `保存并刷新` 负责持久化配置并触发主数据刷新

### 7.4 本地存储策略

用户已明确不希望 App 访问系统钥匙串，因此第一版不使用 Keychain。

设置页相关数据采用本地配置存储即可，不弹钥匙串授权请求。

## 8. 数据来源与接口映射

第一版优先复用 `sub2api` 管理端现有接口与已有页面数据结构，必要时由客户端做二次聚合。

### 8.1 账号监控数据

账号监控页至少需要：

- 账号列表接口：基础账号状态、并发、最近使用
- 使用窗口接口：`5h / 7d` 占用、剩余、重置时间
- 站点统计接口：今日 Token、总 Token、金额字段

### 8.2 仪表盘数据

仪表盘页至少需要：

- 模型维度的聚合数据
- 用户或账号维度的消费排行数据
- 实际金额与标准金额字段

优先策略：

- 优先复用 `sub2api` 网页后台已有仪表盘接口
- 若无直接聚合接口，则客户端从现有管理接口聚合生成所需视图模型

### 8.3 建议 UI 聚合模型

建议前端维护独立展示模型，而不是让 View 直接消费原始 DTO。

建议最少包含：

- `SiteSummary`
  - `siteName`
  - `baseURL`
  - `totalAccounts`
  - `todayTokens`
  - `totalTokens`
  - `todayActualCost`
  - `todayStandardCost`
  - `totalActualCost`
  - `totalStandardCost`
- `AccountCardModel`
  - `id`
  - `name`
  - `state`
  - `currentConcurrency`
  - `lastUsedAt`
  - `fiveHourUtilization`
  - `fiveHourCountdownText`
  - `fiveHourWindowText`
  - `sevenDayUtilization`
  - `sevenDayCountdownText`
  - `sevenDayWindowText`
- `DashboardModelDistributionItem`
  - `modelName`
  - `tokenCount`
  - `actualCost`
  - `ratio`
- `DashboardConsumerRankItem`
  - `displayName`
  - `tokenCount`
  - `actualCost`
  - `standardCost`
  - `primaryModel`

## 9. 排序、状态与筛选规则

### 9.1 正在使用中判断

“正在使用中”采用以下规则：

- `currentConcurrency > 0`
- 或 `lastUsedAt` 距当前时间不超过 `10` 分钟

如果当前没有账号满足条件，则“正在使用中”筛选结果允许为空，不额外制造占位假数据。

### 9.2 高占用阈值

本轮确认后的阈值：

- `5h` 高占用阈值：`60%`
- `7d` 紧张阈值：`80%`

### 9.3 默认排序

默认排序优先级：

1. `inUse`
2. `tight`
3. `healthy`
4. `rateLimited`
5. `unschedulable`

同层级内再按：

1. `5h` 占用更高的优先
2. 名称字母序

该规则用于满足两个目标：

- 正在跑的账号靠前
- 不可用账号不要悬浮到列表顶部

### 9.4 告警表达

用户明确希望“报警不要过于明显展示”，因此状态表达应克制：

- 用状态胶囊和进度颜色表达问题
- 不引入夸张红块、整行闪烁或强提醒横幅

## 10. 交互与刷新策略

建议交互策略：

- 打开弹层后立即刷新
- 手动点击 `刷新` 时立即请求
- 弹层展开期间可定时自动刷新
- 切换 Tab 不改变弹层总宽度

错误处理原则：

- 刷新失败时展示轻量错误提示
- 尽可能保留最近一次成功数据
- 不让弹层因为接口错误变成空白面板

## 11. 视觉风格

视觉风格继续贴近 `sub2api` 后台，但做更适合菜单栏的收敛：

- 白底或极浅灰底
- 细描边
- 扁平化卡片
- 弱阴影或无阴影
- `teal / cyan` 为主强调色
- 状态色仅作点缀
- 避免厚重后台感和装饰性渐变

关键词：

- 扁平
- 紧凑
- 清爽
- 轻管理台
- 信息优先

## 12. 验证目标

实现阶段至少应验证以下结果：

- 弹层保持窄尺寸时，`账号监控` 与 `仪表盘` 都不挤压成异常长条
- `账号监控` 首屏至少能稳定看到两张账号卡
- 顶部汇总卡能同时展示 Token 和金额
- 卡片内 `5h / 7d` 进度、倒计时、窗口摘要信息齐全且不拥挤
- 默认排序符合“正在使用优先、不可用靠后”
- `仪表盘` 在原生布局下能正常展示金额与模型信息
- 设置页可完成 `测试连接` 与 `保存并刷新`
- 设置页不再触发系统钥匙串访问

## 13. 最终结论

第一版最终产品应是一个：

`保持窄弹层尺寸、以账号监控为主、以原生仪表盘为辅的 Sub2API 菜单栏监控器`

它的职责边界已经明确：

- 在菜单栏中快速看整体消耗
- 在菜单栏中快速看每个账号的窗口状态
- 真正需要深度信息时，再统一打开网页端

这让 App 足够轻，也足够实用。
